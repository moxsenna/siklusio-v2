1. Pengantar & Visi Produk

Misi kemanusiaan berbalut keunggulan rekayasa teknologi digital

Problem Statement & Solusi Sistem

Siklusio lahir sebagai solusi komprehensif atas dua tantangan terbesar yang dihadapi oleh wanita usia produktif, khususnya pasangan muda yang sedang merencanakan kehamilan (TTC): manajemen kesehatan reproduksi biologis yang tidak menentu dan perencanaan anggaran persiapan kehamilan/persalinan yang tidak terstruktur.

Siklusio mengawinkan pemantauan biologis berbasis kedokteran obstetri dengan alokasi tabungan kehamilan terarah (pregnancy financial targets). Integrasi ini didesain agar perjalanan fisik ibu menuju kehamilan selaras dengan persiapan finansial keluarga, meminimalkan stressor psikososial selama masa transisi krusial ini.

3 Pilar Utama Siklusio

  • 1
    Universal Health Tracking: Pemantauan fase menstruasi, prediksi masa subur (fertile window), dan ovulasi puncak dengan algoritma medis dinamis berbasis calendar logging yang sangat akurat.
  • 2
    Smart Financial Allocation Sync: Kartu tabungan darurat interaktif kehamilan dengan chip setoran nominal cepat (Quick Set) yang tersinkronisasi dua arah langsung antara Dashboard dan Layar Pengaturan Siklus.
  • 3
    Safe & Moderate Community Queue: Ruang diskusi bebas anonimitas bagi para calon ibu yang dilindungi oleh sistem moderasi konten admin mutakhir berbasis PostgreSQL RPC yang diamankan dari manipulasi client.

2. Topologi & Arsitektur Sistem

Aliran data nir-hambatan lintas client-server modern

Siklusio beroperasi di atas infrastruktur modern dengan pemisahan tanggung jawab yang ketat (*Separation of Concerns*). Melalui pendekatan *single-source-of-truth* pasca eliminasi web client yang berlebih, aplikasi Expo Android/iOS menjadi satu-satunya gerbang interaksi pengguna. Berikut adalah peta interaksi sistem:

1. Client Tier (Universal Expo Hub) Membungkus context terpusat `CycleContext` dan `AuthContext` yang menyatukan alur biologis dan setelan tabungan. Didukung `AsyncStorage` untuk fungsi penayangan luring (offline caching) dan penanganan visual safezone universal pada layout ponsel cerdas.
2. Backend Tier (Express REST Engine) Beroperasi secara mandiri pasca pembongkaran bundel Vite client web. Menyediakan endpoint restfull cerdas yang berinteraksi langsung dengan Google Gemini SDK untuk penyusunan resep makanan ramah anggaran lokal dan analisis audit siklus medis.
3. Cloud Storage & Sec Tier (Supabase) Pusat penyimpanan PostgreSQL relasional dengan pelindung RLS (Row Level Security) yang sangat selektif. Memastikan data demografis medis dan riwayat haid tidak mengalami kebocoran lintas user. Menyimpan fungsi database admin bypass RPC yang atomik.

3. Logika Perhitungan Siklus & Medis

Algoritma kalender biologis dinamis berbasis obstetri klinis

Siklusio menerapkan perhitungan biologis obstetri modern. Kami tidak mengasumsikan panjang siklus yang statis dan kaku. Dengan memindai blok riwayat haid manual (*activity logging*) secara real-time, sistem menyaring penyimpangan biologis dan memetakan rentang masa subur secara dinamis:

Metode Ovulasi Luteal

Hari Ovulasi Puncak

Hari Ovulasi diposisikan tepat 14 hari sebelum hari awal menstruasi siklus berikutnya. Ini didasarkan pada stabilitas klinis fase luteal pasca-ovulasi dalam fisiologi reproduksi wanita.

Ovulasi = PanjangSiklus - 14
Window of Opportunity

Fertile Window (Masa Subur)

Menghitung batas ketahanan sperma hidup dalam rahim (hingga 5 hari) dan ketahanan viabilitas sel telur pasca-ovulasi (24 jam). Rentang fertilitas membentang sepanjang 7 hari penuh.

MasaSubur = [Ovulasi - 5] s.d [Ovulasi + 1]
Medical Guardrail

Batas Aman Abnormalitas

Sistem secara cerdas memangkas data logging abnormal (misal jika pengguna lupa melapor selama berbulan-bulan) dengan batas klinis panjang siklus menstruasi sehat.

20 Hari ≤ Siklus Valid ≤ 45 Hari

Dynamic Period Calculation Logic

Setiap kali pengguna menambahkan laporan "Sedang Menstruasi" secara manual di kalender, sistem akan menghitung blok rentang hari haid yang berurutan. Tanggal pertama dari log menstruasi terdekat akan ditetapkan sebagai HPHT baru, menyinkronkan data profil di database tanpa merusak perhitungan bawaan jika data manual dikosongkan.

5. Desain Skema Database Cloud & RLS

Struktur PostgreSQL relasional terlindung Row Level Security tingkat tinggi

Siklusio memanfaatkan infrastruktur **Supabase Cloud (PostgreSQL)** untuk penyimpanan persisten yang aman. Keamanan privasi data medis sangat dijaga ketat; pengguna biasa tidak dapat membaca data riwayat, catatan harian, maupun alokasi tabungan pengguna lain berkat proteksi **Row Level Security (RLS)** bawaan tabel.

Tabel: `profiles` (Metadata Pengguna)

RLS Active

Menyimpan data demografis dasar, status siklus biologis bawaan, setelan tabungan kehamilan, serta kredensial otorisasi admin.

Aturan Row Level Security (RLS) & Trigger Auto-Profil

Supabase menerapkan trigger PL/pgSQL untuk membuat baris profil pengguna secara otomatis pasca otentikasi pendaftaran berhasil: 

-- Trigger otomatis membuat profile saat akun auth.users dibuat
CREATE OR REPLACE FUNCTION public.handle_new_user() 
RETURNS TRIGGER AS $$
BEGIN
  INSERT INTO public.profiles (id, name, nickname, whatsapp_number, birth_date, children_count)
  VALUES (
    new.id,
    new.raw_user_meta_data->>'name',
    new.raw_user_meta_data->>'nickname',
    new.raw_user_meta_data->>'whatsapp',
    (new.raw_user_meta_data->>'birth_date')::date,
    new.raw_user_meta_data->>'children_count'
  );
  RETURN new;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

CREATE OR REPLACE TRIGGER on_auth_user_created
  AFTER INSERT ON auth.users
  FOR EACH ROW EXECUTE PROCEDURE public.handle_new_user();

mobile-app/src/lib/cycleUtils.ts 
import { differenceInDays, addDays, subDays } from 'date-fns';

export interface ActivityLog {
  date: string;
  isPeriod: boolean;
  symptoms: string[];
}

// Algoritma dinamis obstetri klinis Siklusio
export function calculateCycleState(
  history: ActivityLog[],
  defaultCycle: number = 28,
  defaultPeriod: number = 5
) {
  const periodLogs = history.filter(h => h.isPeriod).sort((a, b) => +new Date(b.date) - +new Date(a.date));
  
  if (periodLogs.length === 0) {
    return { finalCycleLength: defaultCycle, finalPeriodLength: defaultPeriod };
  }

  // Kalkulasi interval antar-blok menstruasi manual
  const intervals: number[] = [];
  let currentBlockStart = new Date(periodLogs[0].date);
  
  for (let i = 1; i < periodLogs.length; i++) {
    const diffDays = differenceInDays(new Date(periodLogs[i - 1].date), new Date(periodLogs[i].date));
    
    // Deteksi jeda blok haid baru (jika tanggal terpisah lebih dari 1.5 hari)
    if (diffDays > 1.5) {
      const interval = differenceInDays(currentBlockStart, new Date(periodLogs[i].date));
      // Guardrail medis: Batasi interval siklus sehat (20 - 45 hari)
      if (interval >= 20 && interval <= 45) {
        intervals.push(interval);
      }
      currentBlockStart = new Date(periodLogs[i].date);
    }
  }

  const finalCycleLength = intervals.length > 0 
    ? Math.round(intervals.reduce((a, b) => a + b, 0) / intervals.length) 
    : defaultCycle;

  return { finalCycleLength, finalPeriodLength: defaultPeriod };
}

8. Keamanan, Protokol RPC & Moderasi Konten

Pemberantasan spam serta ujaran kebencian komunitas melalui saringan admin aman

Supabase Row Level Security (RLS) didesain agar pengguna tidak dapat memanipulasi log, tabungan, maupun menyunting paksa data komunitas secara ilegal dari client side. Ketika ada konten dilaporkan oleh warga komunitas (`community_reports`), admin akan meluncurkan moderasi peninjauan. Siklusio mengintegrasikan fungsi moderasi tangguh berbasis **PostgreSQL RPC (Remote Procedure Call)**:

Protokol Keamanan Moderasi PostgreSQL (`admin_moderate_target`)

Fungsi ini diproteksi dengan klausul `SECURITY DEFINER` untuk memintas batasan RLS secara internal dan aman. Akses bypass hanya diotorisasi jika pengguna teridentifikasi sebagai administrator (`is_admin = true`): 

CREATE OR REPLACE FUNCTION public.admin_moderate_target(
  p_target_type TEXT,
  p_target_id UUID,
  p_action TEXT
)
RETURNS VOID
LANGUAGE plpgsql
SECURITY DEFINER -- Berjalan internal melompati RLS, diisolasi dari modifikasi client
SET search_path = public
AS $$
DECLARE
  v_uid UUID := auth.uid();
  v_is_admin BOOLEAN := public.is_admin(v_uid);
  v_new_hidden BOOLEAN;
  v_new_status TEXT;
  v_report_status TEXT;
BEGIN
  -- Otorisasi ketat
  IF NOT v_is_admin THEN
    RAISE EXCEPTION 'forbidden: admin access required';
  END IF;

  IF p_target_type NOT IN ('post', 'comment') THEN
    RAISE EXCEPTION 'invalid target_type: %', p_target_type;
  END IF;

  IF p_action = 'keep' THEN
    v_new_hidden := FALSE;
    v_new_status := 'kept';
    v_report_status := 'resolved_keep';
  ELSIF p_action = 'remove' THEN
    v_new_hidden := TRUE;
    v_new_status := 'removed';
    v_report_status := 'resolved_hide';
  ELSE
    RAISE EXCEPTION 'invalid action: %', p_action;
  END IF;

  IF p_target_type = 'post' THEN
    UPDATE public.community_posts
       SET is_hidden = v_new_hidden,
           hidden_reason = CASE WHEN v_new_hidden THEN 'admin_action' ELSE NULL END,
           admin_reviewed_at = NOW(),
           admin_review_status = v_new_status
     WHERE id = p_target_id;
  ELSE
    UPDATE public.community_comments
       SET is_hidden = v_new_hidden,
           hidden_reason = CASE WHEN v_new_hidden THEN 'admin_action' ELSE NULL END,
           admin_reviewed_at = NOW(),
           admin_review_status = v_new_status
     WHERE id = p_target_id;
  END IF;

  -- Selesaikan semua antrean laporan terkait
  UPDATE public.community_reports
     SET status = v_report_status,
         resolved_at = NOW(),
         resolver_id = v_uid
   WHERE target_type = p_target_type
     AND target_id = p_target_id
     AND status = 'pending';
END;
$$;
